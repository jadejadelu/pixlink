## 1. 项目概述

### 1.1 核心概念

pixlink 是一个基于 **ZTM (Zero Trust Mesh)** 开源工具构建的游戏联机平台，旨在为普通玩家提供**简单、安全、低延迟**的跨局域网联机体验。

### 1.2 产品形态

- **桌面应用**: 本地运行的跨平台客户端（Windows/macOS/Linux）
- **核心交互**: 用户登陆、创建房间、加入房间、分享游戏、接入游戏
- **附加功能**: 房间内文字聊天、P2P 直连优化

### 1.3 目标用户

- 沙盒游戏玩家（Minecraft、Terraria、Valheim 等）
- 经典局域网游戏爱好者（CS、魔兽争霸、星际争霸等）
- Steam 独立联机游戏玩家
- 小型游戏社区/开黑群组

## 2. 技术基础分析

### 2.1 ZTM 核心能力评估

| 能力维度 | ZTM 特性 | 游戏联机适用性 |
|---------|---------|--------------|
| **网络穿透** | HTTP/2 隧道 + P2P 直连 | ⭐⭐⭐⭐⭐ 自动 NAT 穿透，支持 P2P |
| **安全性** | mTLS 端到端加密 | ⭐⭐⭐⭐⭐ 企业级安全，证书认证 |
| **性能** | C++ 高性能代理 | ⭐⭐⭐⭐⭐ 低延迟，支持 UDP/TCP |
| **跨平台** | x86/ARM/多系统支持 | ⭐⭐⭐⭐⭐ Windows/macOS/Linux/Android |
| **可编程性** | PipyJS 脚本扩展 | ⭐⭐⭐⭐⭐ 可定制游戏专用逻辑 |
| **易用性** | CLI + GUI 桌面应用 | ⭐⭐⭐⭐ 原生提供 GUI 框架 |

### 2.2 系统关键组件设计

服务端pix-server，目前定位是纯后端服务，关键组件包括：

| 名称         | 功能描述                                                         | 技术实现            |
| ---------- | ------------------------------------------------------------ | --------------- |
| pix-server | 用来实现用户数据的处理，包括用户的注册、登陆、登出、注销，ztm客户端证书的颁发和ztm用户的管理，还有用户房间状态信息 | 要求持久化使用数据库mysql |
| ztm-hub    | 用来实现ztm hub能力                                                | 使用开源ztm产品       |
| ztm-agent  | 使用root用户接入ztm hub，用来管理ztm用户证书的颁发                             | 使用开源ztm产品       |


客户端pix-client，目前定位是前端+后端，关键组件包括：

| 名称        | 功能描述                                                                                                                   | 技术实现                                                         |
| --------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| pix-gui   | 1. 用来给用户提供ui界面，用户通过它完成所有ui操作<br>2. 用来和pix-server通信交互，完成用户的注册、登陆、登出、注销请求上报，以及ztm客户端id文件的上传<br>3. 用来实现用户ui请求转化成ztm api调用 | 推荐使用Tauri2.0 + Vue 3 + Vite框架<br>无状态，通过缓存记录非持久化数据，理论上没有持久化数据 |
| ztm-agent | 用来实现接入hub，并实现隧道的开通创建                                                                                                   | 使用开源ztm产品                                                    |



### 2.3 关键技术优势

1. **P2P 直连支持**: ZTM 最新版本已添加 P2P 直接连接功能，可在网络条件允许时绕过 Hub 中转
2. **UDP 隧道**: 支持 `udp/<name>` 隧道，完美适配游戏联机需求
3. **内置应用生态**: zt-tunnel、zt-proxy、zt-terminal、zt-cloud 可直接复用
4. **零配置组网**: 基于证书的身份认证，无需手动配置 IP

## 3. 市场竞争评估

### 3.1 竞品分析矩阵

| 产品 | 价格 | 易用性 | P2P支持 | 跨平台 | 特色功能 | 主要缺陷 |
|-----|------|-------|--------|--------|---------|---------|
| **贝锐蒲公英** | 免费/付费 | ⭐⭐⭐⭐⭐ | ✅ | 全平台 | 企业级SD-WAN | 免费版5人限制 |
| **Tailscale** | 免费/付费 | ⭐⭐⭐⭐ | ✅ | 全平台 | WireGuard基础 | 国内稳定性波动 |
| **ZeroTier** | 免费/付费 | ⭐⭐⭐ | ✅ | 全平台 | 企业级功能 | 配置门槛高 |
| **Radmin VPN** | 完全免费 | ⭐⭐⭐⭐⭐ | ✅ | Windows | 游戏优化 | 仅Windows |
| **HAPPYN** | 免费 | ⭐⭐⭐⭐ | ✅ | 多平台 | 国内优化 | 海外支持弱 |
| **KatoLan** | 免费 | ⭐⭐⭐⭐⭐ | ✅ | Windows | 简洁易用 | 功能较单一 |
| **GameLAN** | 免费 | ⭐⭐⭐⭐⭐ | ✅ | 多平台 | 内置语音 | 新兴工具 |

### 3.2 市场机会分析

**痛点识别**:

1. **现有工具配置复杂**: ZeroTier、Tailscale 需要网络知识
2. **免费版限制多**: 蒲公英5人限制、Hamachi 5人限制
3. **平台支持不全**: Radmin 仅 Windows、部分工具缺少移动端
4. **缺少社交功能**: 大多数工具纯网络层，无聊天/房间管理
5. **开源方案门槛高**: N2N、ZTM 原生需要命令行操作

**机会窗口**:

- 开源技术成熟但缺少**用户友好的包装**
- 游戏联机市场持续增长（沙盒游戏、独立游戏爆发）
- 玩家对**低延迟、P2P直连**需求强烈
- 社区/小团队需要**免费、无人数限制**的解决方案

## 4. 产品定位与差异化

### 4.1 核心定位

> **"基于开源技术的免费游戏联机平台，让异地好友像在同一局域网一样玩游戏"**

### 4.2 差异化策略

| 维度 | pixlink 策略 | 竞品对比 |
|-----|---------------|---------|
| **技术底座** | 基于 ZTM 开源项目，透明可审计 | 竞品多为闭源商业软件 |
| **成本模式** | 基础功能完全免费，无人数限制 | 蒲公英5人、Hamachi5人限制 |
| **易用性** | 三步联机：创建 → 分享 → 加入 | ZeroTier需要配置网络 |
| **社交体验** | 内置房间聊天 + 游戏共享面板 | Radmin无聊天功能 |
| **游戏优化** | 自动识别游戏 + 智能端口映射 | 通用VPN工具无游戏感知 |
| **跨平台** | Windows/macOS/Linux 全支持 | Radmin仅Windows |

### 4.3 价值主张

**对玩家**:
- 🎮 无需网络知识，像开黑一样简单
- 🔒 端到端加密，数据不经过第三方服务器
- ⚡ P2P直连，延迟比中转方案更低
- 💬 房间内聊天，沟通更方便

**对社区**:
- 🆓 免费使用，无成员数量限制
- 🛠️ 开源可定制，支持二次开发
- 🌐 自建 Hub，完全掌控数据

│   ├── 网络隧道
│   │   ├── 自动创建游戏端口映射
│   │   ├── 隧道两端（inbound和outboud，这两个都有可能是多节点）状态显示
│   │   └── 延迟测试与优化
│   └── 游戏共享
│       ├── 共享本地游戏服务端
│       ├── 一键复制连接地址
│       └── 游戏配置模板
│
├── 社交功能
│   ├── 房间聊天
│   │   ├── 文字消息
│   │   ├── 系统通知（加入/离开）
│   │   └── 消息历史
│   └── 语音功能（Future）
│
├── 用户系统
│   ├── 本地身份（基于 ZTM 证书）
│   │   ├── 用户注册（落到pix-server数据库）
│   │   ├── 用户登陆（上传 ztm-agent id文件，服务端生成ztm证书（ztm用户名和pix-server用户名一致），客户端导入证书加入mesh）
│   │   ├── 用户登出（客户端leave mesh）
│   │   ├── 用户关闭软件（客户端停止所有活动进程）
│   │   ├── 用户打开软件（已经登陆过，没有主动注销，本地缓存还在，启动pix-gui，启动ztm-agent，保持上次的房间、隧道等信息（从pix-server获取），执行ztm-agent隧道状态check，并修正）
│   │   ├── 用户打开软件（已经登陆过，主动注销，或者本地缓存丢失，启动pix-gui，启动ztm-agent，同样账户登陆后，保持上次的房间、隧道等信息（从pix-server获取），执行ztm-agent隧道创建并check），不同账户登陆后，重建ztmdb，清理用户数据？？
│   ├── 昵称/头像设置（落库）
│   └── 最近房间记录（缓存+落库）
